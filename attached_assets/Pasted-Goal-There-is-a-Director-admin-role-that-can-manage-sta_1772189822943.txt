Goal:

There is a ‘Director’ (admin) role that can manage staff permissions.

Staff role cannot change roles or permissions.

Avoid ‘main email’ magic checks; use a role field on the user record.

Requirements:

Data model:

StaffUser: id, name, email (unique), passwordHash, role (‘director’ | ‘staff’), isActive, createdAt, lastLoginAt.

Enforce unique email at the database level.

Authentication:

Email + password login.

Store passwords using a strong one-way hash (Argon2id preferred, bcrypt acceptable) with per-user salt.

Do not store plaintext passwords or password reset tokens in logs.

Sessions should be httpOnly, secure cookies; set SameSite=Lax (or Strict if feasible).

Add basic rate limiting to /auth/login.

No user enumeration: login errors should be generic.

Authorization:

All /staff/* routes require authenticated session.

Director-only actions:

View user list

Create user

Deactivate user

Change user role (staff ↔ director)

Staff users can view and edit only their own profile (name, email) but cannot change roles.

“Bootstrap admin” (minimal + safe):

On first run, create one Director account using environment variables:

ADMIN_EMAIL, ADMIN_PASSWORD (or ADMIN_PASSWORD_HASH), ADMIN_NAME.

After bootstrap, Director can create/manage other users via UI.

If Director updates their email, it remains a director because role is stored, not inferred from email.

UI:

Add /staff/admin/users page (Director only).

Table: name, email, role, status (active/inactive), last login.

Controls: Edit role, deactivate, reset password (optional stub), create user.

Add clear confirmation modals for role changes and deactivation.

Security guardrails:

Audit log admin actions (role change, deactivate, create user) with timestamp and actor id (no sensitive payload).

Use CSRF protections if using cookie sessions (or SameSite + CSRF token for unsafe methods).

Validate inputs server-side (email format, password min length, etc.).

Never expose passwordHash in API responses.

Deliverables:

Backend endpoints for user management (director-only)

Minimal UI for managing roles

Tests or manual test checklist: login, role gating, cannot escalate privileges, first-run admin bootstrap works.”
